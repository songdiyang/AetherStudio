# Aether Studio UX 优化 PRD

## 1. 项目信息

| 项目 | 说明 |
|------|------|
| **项目名称** | aether_studio_ux_optimization |
| **技术栈** | Rust + Win32 API + Direct2D（原生 Windows，非 Web） |
| **项目规模** | Cargo Workspace，11 个 crate，106 个 Rust 源文件 |
| **渲染方案** | Direct2D 硬件加速，脏矩形局部重绘 |
| **PRD 类型** | 简单 PRD（不含竞品分析） |

### 原始需求复述

用户反馈编辑器存在大量 UI/UX 及功能问题，特别是文本编辑场景。要求在优化用户体验的同时确保性能不损失、架构保持优秀。经代码探索确认 28 个具体问题，覆盖数据安全（崩溃/数据丢失）、交互体验（焦点管理/滚动跟随）、渲染性能（脏矩形失效/双重渲染）和架构可维护性（巨型文件/状态双重存储）四个维度。

---

## 2. 产品目标

### 目标 1：文本编辑零缺陷 — 消除所有数据丢失与崩溃风险

确保光标移动、撤销/重做、多光标编辑、查找替换等核心编辑操作在 ASCII 和非 ASCII（中文、Emoji、组合字符）文本下均正确工作，不留任何 panic 风险。这是编辑器的生命线，任何数据丢失或崩溃都是不可接受的。

### 目标 2：交互体验流畅自然 — 达到 VS Code 级别的操作直觉

方向键移动自动滚动跟随、焦点切换可预测、状态栏点击精准响应、IME 输入体验流畅。用户不应需要"理解编辑器的焦点逻辑"才能正确输入——焦点在哪里，输入就去哪里。

### 目标 3：渲染性能可控可测 — 让脏矩形优化真正生效

消除双重渲染、全窗口重绘泛滥、无变化的冗余缓存重建等问题。确保脏矩形局部重绘在标签切换、面板折叠等常见场景下实际生效，为 60fps 稳定渲染建立可验证的性能基线。

## 3. 用户故事

### US-1：中文输入安全
> 作为一名使用中文写作的开发者，我希望在代码中混用中英文时，光标移动、单词跳转、删除和查找替换都能正确定位，这样我不会因为字节偏移与字符索引混淆而丢失文本或遇到崩溃。

### US-2：撤销可信赖
> 作为一名频繁编辑代码的开发者，我希望所有编辑操作（包括多光标编辑、全部替换、自动配对）都能通过 Ctrl+Z 撤销，这样我在误操作后总能安全回退，不会因为某些操作"跳过"历史记录而陷入不可逆状态。

### US-3：光标始终可见
> 作为一名浏览长文件的开发者，我希望用方向键移动光标时视图自动滚动跟随，这样光标不会移出可视区域后消失，迫使我手动滚动寻找光标位置。

### US-4：焦点可预测
> 作为一名在编辑器、终端、AI 面板之间切换工作的开发者，我希望点击某个面板后键盘输入自动路由到该面板，切换窗口回来后焦点恢复到我上次操作的位置，这样我不会因为焦点丢失而把按键发到错误的地方。

### US-5：状态栏精准响应
> 作为一名依赖状态栏查看 Git 分支和光标位置的开发者，我希望点击状态栏的每个区域都能触发正确的功能，这样我不会因为索引错位而点到错误的 section。

## 4. 需求池

### P0 — 严重（数据丢失 / 崩溃 / 核心功能失效）

#### REQ-P0-01：修复 `move_cursor_word_left` 字节/字符索引混淆
- **描述**：`cursor_col`（字节偏移）被直接用作 `chars: Vec<char>` 索引，非 ASCII 文本导致光标位置错误甚至越界 panic。需参照 `move_cursor_word_right` 的 `text[..idx].chars().count()` 转换方式修正。
- **影响**：中文/Emoji 文本下单词左移崩溃或光标错位
- **验收标准**：
  - [ ] 在纯中文、中英混排、含 Emoji 的文本上执行 Ctrl+Left 不崩溃
  - [ ] 光标停在正确的单词边界
  - [ ] 与 Ctrl+Right 的行为对称（左移终点 = 右移起点的前一位置）

#### REQ-P0-02：`replace_all` 记录撤销历史
- **描述**：`replace_all` 执行多次 delete+insert 但未调用 `self.history.record()`，替换后 Ctrl+Z 无法撤销。
- **影响**：批量替换不可逆，存在数据丢失风险
- **验收标准**：
  - [ ] 执行 Replace All 后 Ctrl+Z 可完整撤销所有替换
  - [ ] Ctrl+Y 可重做
  - [ ] 撤销后光标恢复到替换前的位置

#### REQ-P0-03：多光标编辑记录撤销历史
- **描述**：`broadcast_insert_char`、`broadcast_delete_char`、`broadcast_insert_newline` 修改缓冲区但不记录 history，多光标编辑后撤销完全失效。
- **影响**：多光标编辑不可逆
- **验收标准**：
  - [ ] 多光标输入字符后 Ctrl+Z 可撤销
  - [ ] 多光标删除字符后 Ctrl+Z 可撤销
  - [ ] 多光标换行后 Ctrl+Z 可撤销
  - [ ] 撤销恢复所有光标位置的原始状态

#### REQ-P0-04：修复 StatusBar 点击检测索引错位
- **描述**：`section_regions()` 用 `.rev()` 推入右 3 项导致枚举索引与 `self.sections` 原始索引不一致，所有状态栏点击路由到错误 section。
- **影响**：状态栏所有交互行为错位
- **验收标准**：
  - [ ] 点击 Git 分支区域触发 Git 分支操作
  - [ ] 点击 UTF-8 区域触发编码操作
  - [ ] 点击光标位置区域触发跳转
  - [ ] 所有 6 个 section 的点击行为与视觉位置一致

#### REQ-P0-05：统一焦点管理（WM_SETFOCUS/WM_KILLFOCUS）
- **描述**：`window_proc` 未处理焦点消息，焦点分散为多个独立布尔/枚举，无统一管理。切换窗口后焦点状态不恢复，键盘输入可能路由到错误面板。
- **影响**：输入丢失或路由到错误面板
- **验收标准**：
  - [ ] `window_proc` 处理 WM_SETFOCUS，恢复上次焦点
  - [ ] `window_proc` 处理 WM_KILLFOCUS，保存当前焦点
  - [ ] 点击编辑器区域 → 焦点回到编辑器，键盘输入到编辑器
  - [ ] 点击终端面板 → 焦点到终端，键盘输入到终端
  - [ ] Alt+Tab 切走再切回 → 焦点恢复到离开前的面板

#### REQ-P0-06：修复脏矩形被全窗口重绘架空
- **描述**：标签切换、面板可见性变化、侧边栏切换均触发 `mark_full_window()`，结合 `infer_from_state` 无变化时默认返回 `FullRedraw`，脏矩形优化在多数交互场景下未生效。
- **影响**：渲染性能退化，无法实现局部重绘收益
- **验收标准**：
  - [ ] 标签切换时仅重绘标签栏 + 编辑区域，不触发全窗口重绘
  - [ ] 面板折叠/展开时仅重绘受影响区域
  - [ ] 光标移动时仅重绘旧行 + 新行（脏矩形）
  - [ ] `infer_from_state` 无状态变化时返回 `None`（跳过渲染）而非 `FullRedraw`
  - [ ] 通过性能日志验证：常见交互的脏矩形面积 < 全窗口面积的 30%

### P1 — 高（重要 Bug / 架构问题）

#### REQ-P1-01：方向键移动光标自动滚动跟随
- **描述**：`move_cursor_up/down` 仅发 `CursorMoved` 事件，不调用 `ensure_cursor_visible`，光标移出可视区后消失。
- **验收标准**：
  - [ ] 方向键下移到最后一行时自动向下滚动
  - [ ] 方向键上移到第一行时自动向上滚动
  - [ ] 滚动后光标始终在可视区域内
  - [ ] PageUp/PageDown 同样保持光标可见

#### REQ-P1-02：`delete_char` 行内退格调用 `emit_edit_events()`
- **描述**：`if cursor_col > 0` 分支只设 `status_message`，未调用 `emit_edit_events()`，导致脏矩形追踪和事件驱动更新缺失。
- **验收标准**：
  - [ ] 退格删除字符后脏矩形正确更新
  - [ ] 事件订阅者收到编辑事件通知
  - [ ] 行内退格与行首退格（跨行删除）行为一致

#### REQ-P1-03：修复脏矩形光标位置用字节偏移当字符列
- **描述**：`cursor_x = ... + self.cursor_col as f32 * char_width()` 将字节偏移直接乘以字符宽度，非 ASCII 文本时脏矩形位置错误。
- **验收标准**：
  - [ ] 中文文本下光标脏矩形位置正确
  - [ ] 无光标残影或撕裂
  - [ ] 混合宽度字符（如全角符号）下脏矩形覆盖完整

#### REQ-P1-04：`replace_all` 替换文本含换行时位置修正
- **描述**：使用预计算的 `find_results` 行号，若 `replace_text` 含 `\n`，行结构变化后后续替换位置全部偏移。
- **验收标准**：
  - [ ] 替换文本含换行符时所有匹配项正确替换
  - [ ] 替换后文件内容与预期一致
  - [ ] 替换后光标位置合理

#### REQ-P1-05：`try_auto_pair` 包裹选区时光标位置更新
- **描述**：插入括号后 `cursor_after` 从未变化的 `self.cursor_col` 计算，history 记录了错误光标位置。
- **验收标准**：
  - [ ] 选中文本后输入 `(` 正确包裹为 `(selection)`
  - [ ] 撤销后光标恢复到选区状态
  - [ ] history 记录的光标位置与实际一致

#### REQ-P1-06：拆分 `window.rs` 巨型文件
- **描述**：3999 行巨型文件，`on_l_button_down` 单函数超 300 行，函数名被刻意拆散为 `on_l_b_u_t_t_o_n_d_o_w_n` 等。
- **验收标准**：
  - [ ] 按职责拆分为独立模块（如 `mouse_handler`、`keyboard_handler`、`window_setup`）
  - [ ] 恢复函数名为正常可读形式
  - [ ] 单文件不超过 800 行
  - [ ] 单函数不超过 80 行
  - [ ] 所有现有功能不受影响

#### REQ-P1-07：消除双重渲染
- **描述**：事件处理分支末尾调用 `state.borrow_mut().render()`，WM_PAINT 也无条件调用 `render()`，同一帧内可能多次完整渲染。
- **验收标准**：
  - [ ] 事件处理中仅标记 dirty，不直接调用 `render()`
  - [ ] 仅 WM_PAINT 触发实际渲染
  - [ ] 同一帧内最多渲染一次
  - [ ] 通过帧计数日志验证无冗余渲染

#### REQ-P1-08：StatusBar 消除魔法索引
- **描述**：`update_git_branch` 用 `get_mut(5)`，`update_cursor_position` 用 `get_mut(2)`，硬编码索引一旦 section 顺序调整即崩溃。
- **验收标准**：
  - [ ] 使用枚举或常量引用 section
  - [ ] section 顺序调整不影响功能
  - [ ] 编译期保证索引安全

#### REQ-P1-09：消除 Tab/EditorState 状态双重存储
- **描述**：`Tab` 和 `EditorState` 持有同名字段，通过 `sync_to_tab`/`sync_from_tab` 手动同步，易不一致。
- **验收标准**：
  - [ ] 状态归属单一化（Tab 持有 EditorState 引用或 EditorState 为唯一源）
  - [ ] 移除手动同步代码
  - [ ] 标签切换后光标、选区、滚动位置正确恢复
  - [ ] 无状态不一致导致的"幽灵选区"或"光标跳跃"

## 5. UI/UX 改进方向

### 5.1 焦点管理体验
建立**单一焦点状态机**，统一管理编辑器、终端、AI 面板、查找面板等所有可聚焦区域。用户操作心智模型简化为："点击哪里，输入到哪里"。窗口失焦时保存焦点快照，重新获焦时精确恢复。

### 5.2 光标可见性保证
所有光标移动操作（方向键、单词跳转、行首行尾、查找跳转）均自动调用 `ensure_cursor_visible`，确保光标在任何操作后始终在可视区域内。滚动行为应有平滑过渡（至少 2-3 行的边距缓冲），避免光标紧贴边缘。

### 5.3 渲染流畅度
消除冗余渲染后，编辑器应能在常规交互（打字、滚动、标签切换）下稳定保持 60fps。脏矩形优化让局部更新真正生效，避免每次按键都重绘整个窗口。用户在 4K 高 DPI 显示器上也不应感到卡顿。

### 5.4 状态栏交互精准化
状态栏作为高频信息入口，每个区域点击行为必须与视觉位置严格对应。Git 分支点击弹出分支列表、编码点击切换编码、光标位置点击跳转行号——所有交互都应即时、准确。

### 5.5 非 ASCII 文本编辑体验
中文、日文、韩文、Emoji 等非 ASCII 文本的编辑体验应与 ASCII 完全一致。光标移动、选区、删除、查找替换在混合文本下均正确工作，不留任何"特殊字符导致 bug"的死角。

## 6. 待确认问题

### Q1：焦点管理架构选择
当前焦点分散为多个独立布尔/枚举。统一方案有两种路径：
- **方案 A**：引入 `FocusManager` 枚举（`Editor`/`Terminal`/`AIPanel`/`Find`/...），集中管理焦点切换
- **方案 B**：保持各面板独立焦点布尔，但增加 `window_proc` 层面的 WM_SETFOCUS/WM_KILLFOCUS 协调

**需要决策**：是否接受方案 A 的重构成本（影响面较广），还是先做方案 B 的最小修复？

### Q2：脏矩形策略 — 渐进优化还是一步到位
脏矩形优化失效涉及多个层面（`mark_full_window` 滥用、`infer_from_state` 默认值、裁剪区域合并）。是否：
- **阶段一**：先修复 `infer_from_state` 默认返回值 + 减少 `mark_full_window` 调用点
- **阶段二**：再做多脏矩形 + 裁剪区域拆分

**需要决策**：是否分阶段交付，还是一次性完成？

### Q3：`window.rs` 拆分粒度
